# easyReview 深度调查 · 第二份：训练理解力的工具版图与教学理念

> 目的：为 easyReview（陌生代码库上手引擎）补齐"如何把普通人训练成能理解整个项目、能 review 代码的人"这一侧，并摸清现存工具版图与空白，避免重复造一个"更漂亮的摘要器"。
> 方法：5 路并行搜索 → 抓取 24 个源 → 抽出 109 条断言 → 对 25 条 top 断言做 3 票对抗式验证（2/3 反驳才判死）→ 合成。**23 confirmed / 2 refuted。**
> 生成时间：2026-07-05。前一份报告《资深程序员如何理解和掌握一段程序》偏"专家如何理解"，本份补"如何训练出这样的人 + 工具版图"。

---

## 执行摘要

现存工具聚成三族，**没有一族真正训练"从看不懂到能 review"的能力**：

1. **导航/相关性覆盖层**（code tour、软件制图如 CodeCity/CodeScene）——能改变新人**看哪里**，但从未被证明能建立**持久的理解或 review 能力**；而且**预设路径会压制探索、制造盲区**。
2. **风险/重要度量化器**（Tornhill 的 churn×complexity 热点、相对代码变更缺陷预测、DOA/Truck-Factor 归属模型）——把 easyReview 的**两个轴都操作化了，且有强实证支撑**。
3. **AI"解释仓库"助手**——**可靠地退化成更好的摘要**，因为仓库级问答受限于模型固有理解力而非喂进去的上下文量，且用户系统性地误解 AI 在做什么。

对 easyReview 核心赌注最强的外部背书有两条：**churn 优先的风险轴**（变更频率作为缺陷预测强于静态复杂度，但必须归一化）和**可执行 ground-truth 验证机制**（Gistify 证明"代码库理解"可被设成运行时可判定的任务，而当前 SOTA 模型恰恰在对应"高风险核心"的长/深执行轨迹上失败）。

**关键空白与警告**：没有任何被调查的工具能可度量地把一个人从"读不懂这个项目"训练到"能 review 它"；预设路径会压制探索、制造盲区；人凭直觉的复杂度指标既不能预测理解在哪断裂、也不能跨代码库迁移——所以地图必须建在**版本控制过程信号 + 可证伪断言检验**上，而不是静态指标的美学或 AI 叙述上。

## 第一部分：既有 onboarding / 代码理解工具，及其为何退化成摘要

**已验证要点**

- **接地的相关性覆盖层确实改变新人导航**（3-0）。Taylor et al. (PPIG 2022, n=15)：CodeTour 组在两个任务无关文件上 0 次激活，非 tour 组则靠文件名线索在其中乱翻。→ 验证 easyReview"接地地图改变你看哪里"的机制。但研究欠功率（无显著结果、8 文件玩具库）。
- **⚠️ 预设路径/地图会压制探索、制造盲区**（3-0）。作者原假设"tour 会拓宽探索"被推翻：tour 用户激活**更少**文件，并警告 tour 可能"劝阻开发者去探索未被覆盖的部分"。**这是本次最重要的设计警告：叙述一条路径 ≠ 建立能力。**
- **软件制图是 easyReview 二维地图的成熟先例**（3-0）。CodeCity 把系统渲染成可导航 3D 城市（类=楼、包=区）。但它只可视化结构，**没有风险轴、没有学习路径**——正是我们的差异点。
- **AI"解释仓库"可靠退化成摘要**（2-1→3-0）。CoReQA (2025-01)：所有受测 LLM 完整度只有 5–6/10，"无论参考上下文多少都无法覆盖问题所有方面"，"随上下文增多略有提升，但终究受限于模型固有能力"（Gemini-1.5：无上下文 5.81 → 全仓 5.94）。安全表述是"无有意义增益"，不是"严格更差"（后者已被证伪）。

**⛏ 对 easyReview**：地图/路径是有效的**导航**层，但**不会自动变成理解**。差异化必须来自我们叠加的**风险轴 + 可验证的能力养成**，且必须**主动对抗盲区**——路径只"建议下一步"，同时要求/奖励学习者在其周围自主觅食，验收要能抓住路径没覆盖的盲点。

**空白**：无工具证明能建立持久理解或 review 能力。

## 第二部分：风险 / 重要度的操作化（直接喂两个轴）

**已验证要点**

- **风险轴 = churn × complexity 热点，变更频率是最强的单一缺陷预测因子**（3-0）。Tornhill：输出"先按变更频率排（最佳预测因子），再按大小"；"代码改动次数比纯大小更能预测缺陷……更复杂的指标很少带来额外预测力"。Rahman & Devanbu 2013 佐证：过程指标 > 代码指标，二者合并增益很小。
- **churn 只有归一化才可靠——绝对变更计数是噪音**（3-0）。Nagappan & Ball (ICSE 2005)："绝对 churn 是差的缺陷密度预测因子，相对 churn 高度可预测"，判别故障/非故障二进制达 89.0% 准确率。⚠️ 单一 Windows Server 2003 案例，89% 是系统特定值不是普适常数。
- **重要度/架构贡献可用归属模型操作化（DOA、Truck/Bus Factor），且信号选择显著影响准确度**（3-0）。AVL 算法算每文件 DOA（创建初始化、自己的提交抬升、他人提交降低，归一化 DOA>0.75 视为作者）。基于提交/DOA 的信号**优于**基于 git-blame 行数的信号（AVL 精确率 Q1 达 1.00，行数法 RIG 仅 0.09）。高 truck-factor 系统准确度会下降。
- **单个静态属性弱，组合成多属性模型才强**（3-0）。Menzies 回顾："静态属性确实重要。单独看可能是弱指标，组合起来能产生超越 SOTA 的强信号。" 叠加"最佳指标随项目而变"→ 应做**按代码库标定的复合模型，而非固定阈值**。
- **人凭直觉的复杂度指标不能预测理解断裂、不跨库迁移**（3-0）。~300 个人定义静态指标的 XGBoost 只到 AUROC 0.63，而无特征的 shadow model（在原始代码+IO 上微调）到 0.86。跨七个数据集"没有任何一个属性在多数数据集里被选中"。→ **风险轴不能建在 McCabe/Halstead/LOC 的美学上并当普适。**

**⛏ 对 easyReview**：
- 风险轴 v1 主信号 = **相对 churn（归一化）+ change coupling**；复杂度作次要；**按仓库标定**的复合分，不用硬阈值。
- 贡献度轴 = **调用图中心度 + 基于提交的 DOA/所有权**（别用行级 blame）。
- 这两个轴我们原本靠"报告 7 条规则 + LLM"接地，现在有了**更硬、更便宜、语言无关（VCS 派生）**的一等信号——尤其利好 Rust/Bevy（不依赖调用图解析）。

## 第三部分：验证机制（可证伪断言 → 撞 ground truth）

**已验证要点**

- **代码库理解可被设成可执行、ground-truth 可判定的任务，而 SOTA 恰在长/深轨迹上失败**（3-0）。Gistify (2025-10)：要求产出一个最小自包含文件、其运行输出必须复现代码库命令的输出——**按运行时匹配判定，不看散文**。SOTA"难以可靠解决，尤其长执行轨迹"（最佳配置 58.7% 保真；最难 30 例 ~43%→21%）。**与 easyReview 直接同构：生成可证伪断言 → 撞 ground truth，难度恰好集中在高风险核心。这是我们验证机制最强的外部背书。**
- **验证应重构为实例级、二元输入-输出一致性判断，而非聚合准确率或叙述**（3-0）。"Beyond Accuracy"(2026-01)：把代码理解"重构为二元输入-输出一致性任务"，"聚合分数掩盖专长，模型选择应实例感知"。→ **验收 = 许多小的可证伪二元检查，而非一个整体的'你懂了没'分数。**

**⛏ 对 easyReview**：这是我们的护城河。每一块（chunk）的验收 = **生成一个可证伪断言**（预测输出 / 预测哪个 Bevy system 会跑 / 预测某组件状态如何变）→ **撞真实运行/测试**。小的二元检查、不是整体评分。这也解释了为什么"AI 解释代码"注定退化：它没有这一步。

## 第四部分：教学理念 & 培养 reviewer（**重大空白**）

**现状：这一侧几乎没有存活的实证断言。**

- **Area 3（学习科学 → 可计算学习路径）：0 条断言通过验证。** 搜索确实surfaced了强先例——**Code Reading Club**（codereading.club，Felienne Hermans《The Programmer's Brain》，固定练习序列刻意训练读码）、**cognitive apprenticeship**——但没有任何"能加速从零到理解整个项目"的可证伪断言扛过对抗验证。**即：把学习科学变成一条可计算、可验收的学习路径，目前没有直接实证背书。**
- **Area 4（培养/度量 reviewer）：仅被单篇 AI 误解论文旁及。** getDX《Competencies for Code Review》操作化了 reviewer 胜任力（27 项能力、105 名 reviewer 四维打分）——是目前最接近"能 review 意味着什么"的度量。但**没有发现任何"教人 review"（而非"帮人 review"）的工具或课程**。
- **用户系统性误解 AI 助手**（2-1→3-0，medium）。ICSE'26：用户对 web 访问、代码执行、非文本输出等有"错位预期"，更深层是"对调试/验证/优化所需信息的误解"。→ AI 生成代码时代，reviewer 的 trust calibration 需要 easyReview 用**验证而非叙述**来对抗。

**⛏ 对 easyReview**：
- **这既是最大空白，也是最大机会**：没人做过可度量地"训练 review 能力"的工具。这就是 easyReview 的独占定位。
- 但**要诚实**：把学习路径当成一个**待验证的假设**（正好接上个人验证计划），拿 Code Reading Club / Programmer's Brain / cognitive apprenticeship 当设计来源，**不要过度宣称**"这套排序科学上被证明有效"。
- 用 getDX 的 27 项能力当"能 review 了"的**验收锚点**——学习路径的终点该能对齐其中若干项。

## 两条被证伪的断言（不得引用为支持）

- ❌ "Guided tour 提升了理解/解决问题能力"——1-2 被推翻。tour 只帮定向，**未转化为真正理解**（7 个 CodeTour 参与者只有 1 个能正确解释 bug 成因）。这**反而支持**我们"叙述会退化成摘要"的论点。
- ❌ "给 LLM 全仓内容比不给严格更差"——0-3 被推翻。安全表述只是"无有意义增益"。

## 悬而未决（建议后续专项研究）

1. **学习科学 → 可计算学习路径**：哪些原则（scaffolding/fading、worked examples/认知负荷、mastery learning、间隔检索、刻意练习、sequencing）对"加速整程序理解"有实证支撑，如何变成可计算、可验收、带进度条的路径？**无存活证据，需专项跟进。**
2. **教 review 能力**：有没有人做过可度量地教 review（而非帮 review）的工具/课程？reviewer 胜任力如何操作化验证？
3. **Rust/Bevy ECS 特异性**：控制流经 ECS 调度器分派而非显式调用图时，churn/DOA 与 Gistify 式运行时验证是否仍表现良好？"执行流"是数据驱动、系统排序的。
4. **揭示 vs 强制觅食的最优平衡**：路径压制探索（发现2）但覆盖层帮导航（发现1）——揭示多少地图/路径 vs 逼学习者自主觅食，才能既建立持久能力又不制造盲区？
